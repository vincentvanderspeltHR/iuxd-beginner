Introduction

Progressive Web Apps are web applications that act as native apps for mobile or pc. This means that you can utilize native features such as camera, push notifications and offline use. On this page we’re going to explain exactly what a Progressive Web App (PWA from now on) is and the advantages of having a PWA.

What is a Progressive Web App?

PWA’s are installable unlike other web applications. Because of this and with the use of manifests PWA’s provide an app-like experience on desktop and mobile. They do this without losing the advantages of being web apps, so you’re able to reach them on any browser through the web. Because of the use of Service Workers (a form of JavaScript workers) they become much faster and very reliable.

PWA’s are fast because they load meaningful content and interactivity on screen much quicker than other web apps. You also don’t have to go through the app store to install it, you can install it smoothly straight from the web. This adds a lot to the overall smooth experience of PWA’s, as installing an app through the app store takes longer and requires more effort from users. As a result, users usually don’t install normal apps as quickly because of all the hassle.

A good tool to check if a webapp is progressive is Lighthouse. Lighthouse is baked into the Chrome DevTools if you press F12 inside Google Chrome. Some good examples of PWA’s are this Offline-Wiki, this Pokédex and a simple Airhorner.

Advantages of a PWA

  • Native app features
  • Installable on PC and mobile
  • Push notifications can be used
  • Fast loading speeds
  • Background Synchronization for updating data

Progressive Web apps are also not very hard to make, and are being pushed by Google. Google believes PWA’s are potentially the future of apps as a whole.

How it works

The main work of a PWA is being done by a JavaScript worker called the Service Worker. The Service Worker is a script that can be run in the background by your browser, separate from your web page. This enables a lot of features PWA’s bring to the table. The main advantage of Service Workers is that it can intercept network requests and handle them according to the app logic.

When a Service Worker first gets installed, it tries to put a list of files you created into the cache so it can potentially get the files from there next time. This depends on the specific cache strategy that has been used. There are multiple strategies that are all viable and have their own pro’s and con’s.

In the diagram you can see how the Service Worker is instantiated for the first time. The service Worker is going to be installed once you go to the website, putting itself in Cache. This happens after you have loaded the page, so that there is no lack of performance on reaching the website. Once the Service Worker is installed, It puts all necessary files from the website into Cache. After this it's ready to be used and activated when reaching the website another time. It does not matter if the user is on-line or off-line at this point, as the Service Worker and the whole website is stored in Cache.

A service worker tries to cleanse old files from the cache on the activate phase, which is just after the install phase. This way, users aren’t stuck with old files which can’t be updated unless they are deleted manually. This step is often overlooked but is actually very important, because this step also makes sure files aren’t cached multiple times. If this step isn’t implemented, the size of the PWA will increase every time a user visits the web app. In the next segment, the most used strategies will be discussed.

PWA strategies

On network response

This strategy is useful for resources that are updated frequently, such as news articles. The service worker first checks if the request matches anything in the cache. If it doesn’t match anything, it’ll try to get it from the network. The page will then be sent to the page and is added to the cache at the same time.

Stale-while-revalidate

The stale-while-revalidate strategy is the opposite of the ‘on network response’ strategy. It is useful in situations where the content isn’t updated a lot and stays the same. It is also useful on frequently updated resources where having the latest version is not essential. This strategy fully utilises the speed of the cache storage. When a service worker gets a request for a page, it will always get the one from the cache and update it later if possible.

On Push message

The Push API is a feature that has been built on top of the Service Worker. This allows the Service Worker to receive a push message which it can turn into a push message on your device if needed. The Push API can 'wake up' your Service Worker in response to a message from the messaging service. This all happens in the background, so not even the tab has to be open. When a push message has been received, it may mean the Service worker needs to receive data from the server, such as a chat message. The Service Worker may then put the received data in the cache and send a push notification to the user's device. This strategy is often implemented next to other strategies, and is obviously useful for chat messaging.

On background-sync

Background sync is another feature that's built on top of the Service Worker. It works almost the same as Push API but on a time schedule instead of a push message. This strategy is useful if you have a lot of non-urgent updates, especially if it happens so regularly that sending a push message each update would be too frequent. News articles are a good example of this.

Cache only

Cache only is a very straight-forward strategy. It is ideal for anything that's static on your site, so stuff that doesn't change much or at all. It's implemented in almost all PWA's since there are always static features on your website.

Network only

This strategy is also very straight-forward. It is used for things that have no offline equivalent like POST-requests. Network only is also used a lot but, unlike cache only, has a workaround. This workaround is background sync.

Cache, falling back to network

This strategy is ideal for building an offline-first web app. It is very similar to ‘stale-while-revalidate’. The only difference is that with this strategy, the cache isn’t updated at all. This does make a difference in speed since it doesn’t have to update the cache in the background. It only gets a request from the network if it is not found in the cache. The opposite of this strategy is ‘Network falling back to cache’.

Cache & network race

In this strategy the service worker lets the cache and network race for the fastest response. If the network gets the page quicker (most of the time this is because the file isn’t in the cache yet) it will show the one from the network, and the other way around. This strategy is ideal for a lot of small assets where you want to get the most performance out of devices with slow disk access.

Network falling back to cache

This is the most commonly used strategy when it comes to an offline experience in your PWA. The Service Worker first tries to get data from the server through the internet. If this fails for some reason (for example, your connection drops) the Service Worker immediately checks if something is available in the cache. If it is, it shows outdated content from the cache. This strategy is on of the quickest and easiest to implement. The one downside of this strategy is that if a user has a very slow connection, but is connected to the internet nonetheless, it won't show content out of the cache first and update it at a later point in time.

Cache then network

Cache then network is one of the fastest strategies there is, and it is even optimised for content that updates frequently. Once the Service Worker gets a page request, it gets it from the server while at the same time showing the user a slightly outdated page from the cache if available. Once this has been done, the Service Worker puts the newest version into the cache and shows the user the updated content later. The only downside here is that the Service Worker is doing much more work then with other strategies. This downside is very negligible since the Service Worker works on it's own thread so it shouldn't affect the device's speed when loading a page.

Generic fallback

The Generic fallback strategy is the easiest to implement, and has been implemented even before PWA's existed. A great example of seeing this strategy in use is user avatars. When Facebook has no connection available and fails to retrieve a user's avatar, it show a basic avatar that is already in the cache. This strategy is great for basic sites that just want to alert the user that there is no internet available without getting the ugly standard 'no internet available'.
